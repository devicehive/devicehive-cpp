89 /**
90 Reports current temperature which is read from the external "device" file.
91 */
92 class TempSensor:
93  public devicehive::Equipment
94 {
95 public:
96  String fileName; ///< @brief The source file name.
97  String lastValue; ///< @brief The last value sent to the server.
98  double margin; ///< @brief The temperature margin.
99 
100 public:
101 
102  /// @brief The shared pointer type.
103  typedef boost::shared_ptr<TempSensor> SharedPtr;
104 
105 
106  /// @brief The factory method.
107  /**
108  @param[in] code The unique identifier.
109  @param[in] name The custom name.
110  @param[in] type The equipment type.
111  @param[in] fileName The source file name.
112  @return The new temperature sensor instance.
113  */
114  static SharedPtr create(String const& code, String const& name,
115  String const& type, String const& fileName)
116  {
117  SharedPtr eq(new TempSensor());
118  eq->code = code;
119  eq->name = name;
120  eq->type = type;
121  eq->fileName = fileName;
122  eq->margin = 0.2;
123  return eq;
124  }
125 
126 public:
127 
128  /// @brief Get the temperature.
129  /**
130  Read the value from the file.
131  There is no error if file doesn't exist.
132 
133  @return The string value.
134  Empty on CRC error.
135  */
136 //! [TempSensor_getValue]
137  String getValue() const
138  {
139  String val;
140 
141  if (!fileName.empty())
142  {
143  std::ifstream i_file(fileName.c_str());
144 
145  String line;
146  while (std::getline(i_file, line))
147  {
148  // check for CRC
149  size_t p = line.find("crc=");
150  if (p != String::npos)
151  {
152  p = line.find("NO", p);
153  if (p != String::npos)
154  return String(); // bad CRC
155  }
156 
157  p = line.find("t=");
158  if (p != String::npos)
159  val = line.substr(p+2);
160  }
161  }
162 
163  // convert to Celcius
164  if (!val.empty())
165  {
166  OStringStream oss;
167  oss << std::fixed << std::setprecision(3)
168  << (boost::lexical_cast<double>(val)/1000.0);
169  val = oss.str();
170  }
171 
172  return val;
173  }
174 //! [TempSensor_getValue]
175 
176 
177  /// @brief Check for really new temperature value.
178  /**
179  @param[in] val The current value.
180  @return `true` if value should be sent to the server.
181  */
182  bool haveToSend(String const& val) const
183  {
184  if (!val.empty())
185  {
186  if (lastValue.empty())
187  return true;
188 
189  const double dt = boost::lexical_cast<double>(val)
190  - boost::lexical_cast<double>(lastValue);
191  return fabs(dt) > margin;
192  }
193 
194  return false;
195  }
196 };

646 
647 This simple example aims to demonstrate usage of DeviceHive framework from C++
648 application.
649 
650 In terms of DeviceHive framework this example is **Device** implementation:
651 
652 ![DeviceHive framework](@ref preview2.png)
653 
654 Our device is one of small computers (like [BeagleBone] or [RaspberryPi])
655 which runs Linux and manages the following peripheral:
656  - simple LED, allowing user to switch it *ON* or *OFF*
657  - DS18B20 sensor, providing user with temperature measurements
658 
659 The Internet connection is provided by 3G modem connected to device's USB port.
660 
661 Android application is used as a **Client** to control LED state and to monitor
662 temperature.
663 
664 Please see @ref page_start to get information about how to prepare toolchain
665 and build the *boost* library.
666 
667 
668 Application
669 ===========
670 
671 Application is a console program and it's represented by an instance of
672 simple_dev::Application. This instance can be created using
673 simple_dev::Application::create() factory method. Application does all its
674 useful work in simple_dev::Application::run() method.
675 
676 ~~~{.cpp}
677 using namespace simple_dev;
678 int main(int argc, const char* argv[])
679 {
680  Application::create(argc, argv)->run();
681  return 0;
682 }
683 ~~~
684 
685 Application contains the following key objects:
686  - simple_dev::Application::m_ios the IO service instance to perform various
687  asynchronous operations (such as HTTP requests and update timers)
688  - simple_dev::Application::m_device the DeviceHive device object which
689  contains several equipment objects:
690  - any number of simple_dev::LedControl instances
691  - any number of simple_dev::TempSensor instances
692  - simple_dev::Application::m_api the "clue" which links our application and
693  the DeviceHive server
694 
695 
696 LED control
697 -----------
698 
699 We assume that actual LED hardware is mapped by kernel to some *device* file
700 which is in virtual filesystem such as `/proc` or `/sys`. Such *device* file
701 name based on the GPIO pin number. For example, the `gpio38` pin
702 on [BeagleBone] is mapped to the `/sys/class/gpio/gpio38/value` *device* file.
703 Also [BeagleBone] has four default LED controls which are placed on the board
704 and have the following *device* files:
705  - `/sys/class/leds/%beaglebone::usr0/brightness`
706  - `/sys/class/leds/%beaglebone::usr1/brightness`
707  - `/sys/class/leds/%beaglebone::usr2/brightness`
708  - `/sys/class/leds/%beaglebone::usr3/brightness`
709 
710 We bind simple_dev::LedControl instance with a *device* file. Once we've got
711 command from the server to change the LED state, this state
712 will be immediately written to the corresponding *device* file
713 using simple_dev::LedControl::setState() method, which is very simple:
714

718 Temperature sensor
719 ------------------
720 
721 The temperature sensor also uses *device* file. But this file is written by
722 hardware and simple_dev::TempSensor instance reads it periodically.
723 The *device* file format is not so simple as for LED control but we can extract
724 actual temperature value searching for the `t=` keyword.
725 
726 Also we have to verify integrity of the *device* file and skip measurement if
727 CRC equals to "NO".
728 
729 @snippet examples/simple_dev.hpp TempSensor_getValue
730 
731 Typical DS18B20 file content is:
732 
733 ~~~
734 83 01 4b 46 7f ff 0d 10 66 : crc=66 YES
735 83 01 4b 46 7f ff 0d 10 66 t=24187
736 ~~~
737 
738 It means that current temperature is 24.187C.
739 
740 Our application periodically checks all the connected temperature sensors and
741 sends notifications to the DeviceHive server if the temperature measurement has
742 been changed (more than 0.2 degree) since last time.
743 
744 
745 Control flow
746 ------------
747 
748 At the start we should register our device on the server using
749 simple_dev::Application::asyncRegisterDevice() method. If registration is
750 successful we start listening for the commands from the server using
751 simple_dev::Application::asyncPollCommands() method. At the same time we start
752 *update* timer and periodically check the temperature sensors using
753 simple_dev::Application::asyncUpdateSensors() method. If temperature
754 measurement has been changed, then we send notification to the DeviceHive
755 server.

758 Command line arguments
759 ----------------------
760 
761 Application supports the following command line arguments:
762 
763 | Option | Description
764 |-----------------------------------------|-----------------------------------------------
765 | `--deviceId <id>` | `<id>` is the device identifier
766 | `--deviceName <name>` | `<name>` is the device name
767 | `--deviceKey <key>` | `<key>` is the device authentication key
768 | `--networkName <name>` | `<name>` is the network name
769 | `--networkKey <key>` | `<key>` is the network authentication key
770 | `--networkDesc <desc>` | `<desc>` is the network description
771 | `--deviceClassName <name>` | `<name>` is the device class name
772 |`--deviceClassVersion <ver>` | `<ver>` is the device class version
773 | `--server <URL>` | `<URL>` is the server URL
774 | `--led <id> <name> <file>` | `<id>` is the unique identifier \n `<name>` is the LED control name \n `<file>` is the *device* file name
775 | `--temp <id> <name> <file>` | `<id>` is the unique identifier \n `<name>` is the temperature sensor name \n `<file>` is the *device* file name
776 
777 The `--led` and `--temp` commands can be used to create any number of equipment
778 instances (LED controls and temperature sensors respectively).
779 
780 For example, to start application on [BeagleBone] board run the following command:
781 
782 ~~~{.sh}
783 ./simple_dev --led p8_3 "LED" "/sys/class/gpio/gpio38/value" \
784  --temp p8_6 "DS18B20" "/sys/bus/w1/devices/28-00000393268a/w1_slave"
785 ~~~
786 
787 Where `gpio38` is the name of output GPIO pin which is used to control LED.
788 And `28-00000393268a` is the name of *1-wire* device.
